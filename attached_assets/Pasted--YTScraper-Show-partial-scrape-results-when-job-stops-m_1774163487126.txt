# YTScraper — Show partial scrape results when job stops mid-way
# Paste into Replit Agent
# Only fix the scrape page and jobs page comment explorer visibility logic

---

## THE PROBLEM

When a scrape stops mid-way (credits ran out), the job status is
NOT 'completed' — it might be 'cancelled', 'failed', or stuck on 'running'.

The Comment Explorer currently only shows when status === 'completed'.
So even though 445 comments exist in the database, the user sees nothing.

Fix: Show the Comment Explorer whenever downloaded_comments >= 100,
regardless of the job status.

---

## FIX 1 — Scrape page: transition to Comment Explorer based on comment count

Find the scrape page state logic. Currently it transitions to State C
(Comment Explorer) only when status === 'completed'.

Change the condition everywhere from:

```typescript
// OLD — only shows on completed
if (job.status === 'completed') {
  setScrapeState('completed')
}
```

To this:

```typescript
// NEW — show comments if we have enough data, regardless of status
const hasPartialData = (job.downloaded_comments ?? 0) >= 100

if (job.status === 'completed') {
  setScrapeState('completed')
  toast.success(`✓ Scraped ${job.downloaded_comments?.toLocaleString()} comments!`)
}
else if (job.status === 'failed' || job.status === 'cancelled') {
  if (hasPartialData) {
    // We have data — show it instead of the error screen
    setScrapeState('completed')  // reuse the completed state to show explorer
    setActiveJob(job)
    toast.warning(
      `Scrape stopped at ${job.downloaded_comments?.toLocaleString()} comments (${
        job.status === 'cancelled' ? 'cancelled' : 'credits ran out'
      }). Showing available data.`,
      { duration: 6000 }
    )
  } else {
    // Less than 100 comments — show the error/failure screen
    setScrapeState('failed')
    toast.error(job.error_message ?? 'Scrape stopped before any data was collected.')
  }
}
```

## FIX 5 — Also fix: stuck "running" jobs on page load

If a job is stuck in 'running' status (edge function crashed mid-scrape),
the page currently shows the progress card forever with no way out.

Add a timeout: if a job has been 'running' for more than 10 minutes
AND has downloaded_comments >= 100, treat it as partial-complete:

```typescript
// In the job polling / page load check:
const isStuck = (
  job.status === 'running' &&
  job.started_at &&
  (Date.now() - new Date(job.started_at).getTime()) > 10 * 60 * 1000 // 10 min
)

if (isStuck && (job.downloaded_comments ?? 0) >= 100) {
  // Auto-mark as cancelled so it shows in the explorer
  await supabase.from('jobs').update({
    status: 'cancelled',
    completed_at: new Date().toISOString(),
    error_message: 'Job timed out — showing partial results',
  }).eq('id', job.id)
}
```

## SUMMARY OF CHANGES

1. Scrape page: show Comment Explorer when downloaded_comments >= 100,
   even if status is 'failed' or 'cancelled' — with amber "partial results" banner

2. Jobs list: show "View X Comments (partial)" button for any job with >= 100 comments

3. Job detail page: render Comment Explorer for any job with >= 100 comments

4. Stuck running jobs: auto-cancel after 10 min and show partial results

The threshold of 100 comments is the minimum meaningful dataset.
Below 100 = show error. At or above 100 = show what we have.
